# Git Commit Skill

自动分析代码变更并生成规范的 Git 提交信息。

## 何时使用

- 用户要求提交代码、commit、提交改动
- 用户要求生成 commit message
- 用户说"提交一下"、"帮我 commit"等
- 用户提到 push、推送代码
- 用户要求合入某个分支、merge 到某分支

## 提交模式

本 skill 支持两种模式：

| 模式 | 格式 | 何时使用 |
|------|------|----------|
| 简洁模式（默认） | `<type>(<scope>): <subject>` | 默认使用，单行提交信息 |
| 详细模式 | `<type>(<scope>): <subject>` + body | 用户明确要求"详细提交"、"加上 body"、"详细说明改动"时使用 |

如果用户没有指定模式，使用简洁模式。

## 执行流程

### 第一步：检查工作区状态

运行 `git status` 查看当前变更。如果工作区干净（无任何变更），告知用户没有需要提交的内容并停止。

### 第二步：分析变更内容

1. 运行 `git diff --cached` 查看已暂存的变更
2. 运行 `git diff` 查看未暂存的变更
3. 如果没有已暂存的文件但有未暂存的文件，询问用户：
   - 是否暂存全部文件（`git add .`）
   - 还是指定要暂存的文件
4. 等用户确认后执行暂存操作

### 第三步：生成提交信息

根据 `git diff --cached` 的内容生成提交信息。

#### Type 选择规则

根据变更内容选择最准确的 type：

| Type | 使用场景 |
|------|----------|
| `feat` | 新增功能、新增文件实现新特性 |
| `fix` | 修复 Bug、修正错误行为 |
| `docs` | 仅文档变更（README、注释、JSDoc 等） |
| `style` | 代码格式调整，不影响逻辑（空格、分号、换行等） |
| `refactor` | 重构代码，既非新功能也非修复 |
| `perf` | 性能优化 |
| `test` | 添加或修改测试 |
| `build` | 构建系统或外部依赖变更（webpack、npm 等） |
| `ci` | CI/CD 配置变更 |
| `chore` | 其他杂项（.gitignore、配置文件等） |

#### Scope 推断规则

- 从变更文件的路径推断模块名，例如 `src/auth/login.ts` → scope 为 `auth`
- 如果变更跨多个模块，使用最主要的模块或省略 scope
- scope 使用英文小写

#### Subject 规则

- 使用中文描述
- 简洁明了，不超过 50 个字符
- 不以句号结尾

#### Body 规则（仅详细模式）

- 使用中文
- 列出关键变更点，每条以 `- ` 开头

### 第四步：确认并提交

1. 列出本次提交涉及的文件，并对每个文件用一句话概括修改内容
2. 将生成的提交信息一并展示给用户

展示格式示例：

```text
涉及文件：
| 文件 | 修改概要 |
|------|----------|
| src/auth/login.ts | 新增用户登录接口 |
| src/auth/middleware.ts | 添加 JWT 认证中间件 |

提交信息：
feat(auth): 添加用户登录功能
```

3. 询问用户是否确认，或是否需要修改
4. 用户确认后执行：
   - 简洁模式：`git commit -m "<type>(<scope>): <subject>"`
   - 详细模式：`git commit -m "<type>(<scope>): <subject>" -m "<body>"`

### 第五步：推送（仅当用户提到 push/推送时）

如果用户在请求中提到了 push 或推送：

1. 先执行 `git pull --rebase` 拉取远程最新代码并 rebase
2. 如果 rebase 产生冲突，停止操作，告知用户冲突文件列表，等待用户手动解决
3. rebase 成功后执行 `git push`
4. 如果当前分支没有上游追踪分支，使用 `git push -u origin <branch>`

### 第六步：合并到目标分支（仅当用户提到合入/merge 某分支时）

如果用户要求将当前分支合入某个目标分支（如 main、develop）：

1. 确保当前分支的改动已提交且已推送
2. 切换到目标分支：`git checkout <target>`
3. 拉取目标分支最新代码：`git pull --rebase`
4. 合并当前分支：`git merge <source-branch>`
5. 如果产生冲突，停止操作，告知用户冲突详情，等待用户解决
6. 合并成功后推送目标分支：`git push`
7. 切换回原来的工作分支：`git checkout <source-branch>`

### 第七步：拆分建议

如果检测到变更涉及多个不相关的改动（例如同时有新功能和 Bug 修复），主动建议用户拆分为多次提交，并给出每次提交应包含的文件。

## 示例

### 简洁模式（默认）

```
feat(auth): 添加用户登录功能
```

```
fix(cart): 修复购物车数量计算错误
```

```
docs: 更新 README 安装说明
```

### 详细模式

```
feat(auth): 添加用户登录功能

- 实现基于 JWT 的用户认证
- 添加登录接口和中间件
- 新增用户模型定义
```

## 注意事项

- 你必须在用户明确确认后才能执行 `git commit`
- 推送和合并操作仅在用户明确提到时才执行，不要主动发起
- 执行 push 前必须先 `git pull --rebase`，保持提交历史线性
- 遇到 rebase 或 merge 冲突时立即停止，展示冲突文件，等用户解决
- 如果用户要求修改生成的提交信息，按用户要求调整后再次确认
